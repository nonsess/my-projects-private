# Изоком — потеря контекста в боте

## Суть

Задача #28: бот не помнит контекст в Telegram. Оказалось — проблема воспроизводится и на веб-клиенте, т.к. бэкенд один.

## Гипотезы (от сессии 2026-05-21)

- **A** — неправильный `thread_id` в TG-адаптере → не объясняет проблему на вебе, снята
- **B** — `State.messages` объявлен без `add_messages` reducer → каждый вызов ноды перезаписывает список сообщений, а не добавляет → **приоритет диагностики**
- **C** — `completeness_check` смотрит только на `last_user_message`, не на весь диалог → тоже кандидат

## Корень проблемы (выяснено 2026-05-22)

Redis checkpointer и `add_messages` reducer работали корректно — state накапливался. Проблема была в **форматировании при вызове LLM**:
- `completeness_check`: история склеивалась в плоскую строку и отправлялась одним user-сообщением
- `general_response`: передавал только последнее сообщение пользователя, без истории
- `router`: только `last_human.content`, без предыдущего контекста

## Фикс (2026-05-22)

Все три ноды переведены на чередующиеся роли `user`/`assistant`:
- `completeness_check.py`: `messages[-10:]` → `list[{role, content}]`
- `general_response.py`: `messages[-6:]` → `list[{role, content}]`
- `router.py`: `messages[-2:]` → `list[{role, content}]`

Закоммичено и запушено: commit `359a07b` → `brain-boost-academy/new-izokom`.

## Проверка

Протестировано на живом стеке (Docker + host Ollama + Redis + Qdrant) через WebSocket:
- Все 5 параметров сразу → follow-up без повтора → бот не переспросил ✓
- Неполные параметры → уточнение → добавил недостающее → бот собрал из обоих ходов ✓
